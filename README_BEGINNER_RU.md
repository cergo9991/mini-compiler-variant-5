# Мини-компилятор: подробное объяснение для новичка

Этот файл дополняет основной `README.md` и объясняет проект простыми словами: что делает каждая часть компилятора, где она находится в репозитории и как файлы взаимодействуют друг с другом.

## 1. Что вообще делает этот проект

Проект принимает программу на небольшом C-подобном языке и последовательно проходит несколько этапов:

1. Читает текст программы.
2. Разбивает текст на токены: ключевые слова, идентификаторы, числа, знаки операций.
3. Проверяет, что токены образуют корректную программу по правилам грамматики.
4. Строит AST, то есть абстрактное синтаксическое дерево.
5. Выполняет семантический анализ: проверяет типы, области видимости, корректность вызовов функций и `return`.
6. Генерирует LLVM IR.
7. По LLVM IR выпускает объектный файл под MIPS.
8. Объектный файл можно слинковать с `runtime/main.c` и запустить.

Коротко:

- `lexer.l` отвечает за "какие куски текста мы видим".
- `parser.y` отвечает за "как эти куски складываются в программу".
- `ast.hpp` описывает "в каком виде хранить структуру программы".
- `sema.cpp` отвечает за "имеет ли программа смысл".
- `codegen.cpp` отвечает за "как превратить структуру программы в LLVM IR и объектный код".
- `driver.cpp` собирает все этапы в один конвейер.

## 3. Что такое лексика и за что она отвечает

Лексика, или лексический анализ, это самый первый этап. На нем компилятор смотрит на обычный текст и превращает его в токены.

Например, строка:

```c
int x = 10 + 2;
```

превращается примерно в такую последовательность:

- `INT_KW`
- `IDENT("x")`
- `'='`
- `INTEGER(10)`
- `'+'`
- `INTEGER(2)`
- `';'`

В этом проекте лексика реализована в файле `src/lexer.l`.

Что именно он делает:

- пропускает пробелы и переводы строк;
- пропускает комментарии `// ...` и `/* ... */`;
- распознает ключевые слова `return`, `if`, `else`, `for`, `do`, `while`, `fn`, `int`;
- распознает составные операторы `==`, `!=`, `<=`, `>=`, `&&`, `||`, `->`;
- распознает числа;
- распознает идентификаторы;
- для остальных одиночных символов возвращает сам символ, например `'+'`, `'-'`, `'('`, `')'`, `';'`.

Почему это важно:

- парсеру гораздо проще работать не с сырым текстом, а с токенами;
- на этапе лексики определяются позиции символов в файле, чтобы потом показывать ошибки с номером строки и столбца.

## 4. Что такое грамматика и за что она отвечает

Грамматика описывает, какие последовательности токенов считаются правильной программой.

Пример идеи:

- функция состоит из типа, имени, списка параметров и блока;
- блок состоит из списка операторов;
- выражение может состоять из присваивания, арифметики, сравнений, вызова функции и так далее.

В проекте грамматика реализована в `src/parser.y`.

Именно там записаны правила вроде:

- `function`
- `statement`
- `expression`
- `additive`
- `multiplicative`
- `primary`

То есть грамматика отвечает не за смысл программы, а за ее форму.

Пример:

```c
int compiled_fn(int a) {
    return a + 1;
}
```

С точки зрения грамматики это корректно, потому что:

- есть функция;
- у функции есть тело;
- внутри тела есть `return`;
- в `return` стоит корректное выражение.

Но если написать:

```c
int compiled_fn(int a) {
    return if;
}
```

это уже синтаксическая ошибка, потому что токены не подходят ни под одно правило выражения.

## 5. Что такое AST

AST, или Abstract Syntax Tree, это абстрактное синтаксическое дерево. Это внутренняя структура данных, в которую парсер складывает разобранную программу.

Почему дерево называется "абстрактным":

- в нем уже нет лишних деталей исходного текста;
- в нем хранятся не символы, а смысловые узлы: функция, блок, `if`, `for`, литерал, бинарная операция, вызов функции.

Пример:

```c
return a + 2;
```

может быть представлен так:

- `ReturnStmt`
- внутри него `BinaryExpr(Add)`
- слева `VarExpr("a")`
- справа `IntLiteral(2)`

В этом проекте AST описан в `src/ast.hpp`.

Какие узлы там есть:

- выражения:
  - `IntLiteral`
  - `VarExpr`
  - `UnaryExpr`
  - `BinaryExpr`
  - `AssignExpr`
  - `TernaryExpr`
  - `CallExpr`
- операторы:
  - `BlockStmt`
  - `VarDeclStmt`
  - `ExprStmt`
  - `ReturnStmt`
  - `IfStmt`
  - `ForStmt`
  - `DoWhileStmt`
- верхний уровень:
  - `FunctionDecl`
  - `Program`

AST нужен потому, что и семантика, и генерация кода работают уже не с текстом, а с понятной структурой программы.

## 6. Что такое семантика и за что она отвечает

Семантика отвечает на вопрос: "программа не только правильно записана, но и имеет ли смысл?"

Это уже не проверка формы, а проверка корректности использования сущностей.

Примеры семантических ошибок:

- использование необъявленной переменной;
- попытка присвоить значение неправильного типа;
- вызов несуществующей функции;
- передача неправильного числа аргументов;
- отсутствие обязательного `return`;
- условие `if`, `for`, `do/while`, `!`, `&&`, `||` не булево.

В проекте семантика находится в:

- `src/sema.hpp`
- `src/sema.cpp`

Что делает семантический анализатор:

- собирает таблицу функций;
- проверяет, что существует `compiled_fn`;
- проверяет сигнатуру `compiled_fn`: `int compiled_fn(int arg)`;
- ведет таблицы областей видимости переменных;
- вычисляет тип каждого выражения и записывает его в `Expr::inferredType`;
- проверяет типы операторов и выражений;
- проверяет корректность `return`;
- формирует понятные сообщения об ошибках с координатами.

Важно:

- синтаксис говорит "так писать можно";
- семантика говорит "это имеет смысл";
- генерация кода запускается только после успешной семантики.

## 7. Что такое генерация кода

После AST и семантики компилятор переходит к генерации кода.

В этом проекте генерация идет в два шага:

1. Из AST строится LLVM IR.
2. Из LLVM IR выпускается объектный файл под MIPS.

Реализация находится в:

- `src/codegen.hpp`
- `src/codegen.cpp`

Что делает генератор:

- создает LLVM `Module`;
- объявляет функции;
- создает LLVM basic blocks;
- создает инструкции для арифметики, сравнений, `if`, `for`, `do/while`, вызовов функций, `return`;
- умеет сохранить текстовый LLVM IR;
- умеет выпустить объектный файл под target triple `mipsel-unknown-linux-gnu`.

## 8. Что такое LLVM IR простыми словами

LLVM IR это промежуточное представление программы.

Оно нужно, чтобы:

- не генерировать машинный код вручную сразу из AST;
- получить удобную и стандартную промежуточную форму;
- использовать готовую инфраструктуру LLVM для проверки и выпуска кода под конкретную архитектуру.

Можно думать так:

- исходный учебный язык - это язык высокого уровня;
- LLVM IR - это более низкоуровневое, но еще читаемое представление;
- объектный файл MIPS - это уже код для целевой платформы.

LLVM IR удобно тем, что:

- оно строго типизировано;
- из него хорошо видны блоки переходов и вычисления;
- LLVM умеет проверять корректность модуля и функций;
- на его основе можно выпускать код под разные архитектуры, если есть target backend.

## 9. Как файлы взаимодействуют друг с другом

Полный поток данных выглядит так:

```text
Исходный .mc файл
    ->
driver.cpp
    ->
lexer.l
    ->
parser.y
    ->
AST (ast.hpp)
    ->
sema.cpp
    ->
codegen.cpp
    ->
LLVM IR (.ll, если включен --emit-ir)
    ->
Object file (.o) под MIPS
    ->
runtime/main.c + cross-compiler
    ->
Исполняемый MIPS-файл
```

Теперь более подробно.

### `driver.cpp`

Это координатор всего конвейера.

Он:

- открывает входной файл;
- запускает `yyparse()`;
- получает построенную программу из глобального `g_parsed_program`;
- запускает `SemanticAnalyzer`;
- запускает `CodeGenerator`;
- по опциям сохраняет `.ll` и `.o`.

### `lexer.l`

Общается с `parser.y`.

Он не строит AST и не знает про смысл программы. Его задача только вернуть очередной токен и заполнить значение токена, например число или имя идентификатора.

### `parser.y`

Общается:

- с `lexer.l`, потому что получает от него токены;
- с `ast.hpp`, потому что создает AST-узлы;
- с `driver.cpp`, потому что после разбора результат попадает в `g_parsed_program`.

### `ast.hpp`

Это общий язык общения между этапами.

С ним работают:

- `parser.y`, когда создает дерево;
- `sema.cpp`, когда проверяет дерево;
- `codegen.cpp`, когда генерирует LLVM IR по дереву.

### `sema.cpp`

Получает AST от парсера и:

- добавляет сведения о типах;
- проверяет таблицы имен;
- не выпускает программу дальше, если в ней ошибки.

После него AST уже содержит важную информацию, например `inferredType` у выражений.

### `codegen.cpp`

Получает AST после успешной семантики и строит LLVM IR.

Он особенно зависит от семантики, потому что использует уже вычисленные типы выражений, например при загрузке переменной или построении `PHI` для тернарного оператора.

### `runtime/main.c`

Это не часть компилятора как такового, а минимальная программа для запуска результата.

Она ожидает, что скомпилированная программа даст функцию:

```c
int64_t compiled_fn(int64_t arg);
```

и вызывает ее с аргументом `10`.

На текущем уровне проект поддерживает:

## 11. Как понять разницу между этапами на одном примере

Возьмем код:

```c
int compiled_fn(int a) {
    int x = a + 1;
    if (x > 10) {
        return x;
    }
    return 0;
}
```

Что происходит:

### Лексика

Текст превращается в токены:

- `INT_KW`
- `IDENT(compiled_fn)`
- `'('`
- `INT_KW`
- `IDENT(a)`
- `')'`
- `'{'`
- ...

### Синтаксис

Парсер понимает, что это:

- функция;
- внутри блока объявление переменной;
- затем `if`;
- затем `return`.

### AST

Строится дерево:

- `Program`
- `FunctionDecl`
- `BlockStmt`
- `VarDeclStmt`
- `IfStmt`
- `ReturnStmt`

### Семантика

Проверяется:

- что `a` объявлена;
- что `a + 1` допустимо;
- что `x > 10` дает `bool`;
- что `if` получает `bool`;
- что `return x` и `return 0` соответствуют типу функции `int`.

### Генерация кода

Строятся LLVM basic blocks и инструкции:

- выделение памяти для `a` и `x`;
- вычисление `a + 1`;
- сравнение `x > 10`;
- условный переход;
- возврат результата.

## 12. Чем отличаются синтаксические и семантические ошибки

Синтаксическая ошибка:

```c
int compiled_fn(int a) {
    return ;
}
```

Проблема в форме записи: после `return` ожидается выражение.

Семантическая ошибка:

```c
int compiled_fn(int a) {
    if (a) {
        return 1;
    }
    return 0;
}
```

В этом проекте `if` требует `bool`, а `a` имеет тип `int`, поэтому форма записи правильная, но смысл с точки зрения правил языка неправильный.

## 14. Ответы на вопросы по проекту

### 1. Как в коде реализованы правила грамматики? Что делает код с `$$`?

Правила грамматики реализованы в файле `src/parser.y` с помощью Bison.

Каждое правило имеет вид:

```bison
nonterminal
    : часть_1 часть_2 ...
      {
          semantic action
      }
    ;
```

Например, правило функции:

```bison
function
    : type IDENT '(' param_list_opt ')' block
      {
          ...
          $$ = new mc::FunctionDecl(...);
      }
```

означает:

- если парсер увидел `type IDENT ( ... ) block`,
- значит это функция,
- и в действии нужно создать AST-узел `FunctionDecl`.

Что значат специальные обозначения Bison:

- `$$` - значение, которое будет результатом всего правила;
- `$1` - значение первого символа справа;
- `$2` - значение второго символа справа;
- `$3` - значение третьего символа справа и так далее.

Пример:

```bison
additive
    : additive '+' multiplicative
      {
          $$ = new mc::BinaryExpr(mc::BinaryOp::Add,
                                  std::unique_ptr<mc::Expr>($1),
                                  std::unique_ptr<mc::Expr>($3));
      }
```

Здесь:

- `$1` - левое выражение;
- `$2` - токен `'+'`, он отдельно в действии почти не нужен;
- `$3` - правое выражение;
- `$$` - результат, то есть новый узел AST для сложения.

Важно:

- этот код называется `semantic actions` у Bison;
- это не семантический анализ типов;
- это действия, которые выполняются в момент синтаксического разбора и обычно строят AST.

### 2. Как и где происходит генерация кода под целевую платформу

Генерация кода происходит в `src/codegen.cpp`.

Процесс такой:

1. Из AST строится LLVM IR.
2. LLVM IR проверяется.
3. LLVM выпускает объектный файл под MIPS.

Ключевые места:

- `CodeGenerator::generate()` создает LLVM-модуль и генерирует IR для всех функций;
- `CodeGenerator::emitIR()` сохраняет текстовый `.ll` файл;
- `CodeGenerator::emitObject()` создает `TargetMachine` и выпускает `.o`.

Целевая платформа задается строкой:

```cpp
std::string targetTriple_ = "mipsel-unknown-linux-gnu";
```

Это означает, что итоговый объектный файл строится под MIPS little-endian Linux ABI.

Принцип работы:

- `declareFunction()` объявляет LLVM-функции;
- `codegenFunction()` создает `entry`-блок и генерирует тело;
- `codegenStmt()` обрабатывает операторы;
- `codegenExpr()` обрабатывает выражения;
- `emitObject()` передает LLVM-модуль backend-у MIPS.

После этого объектный файл линкуется с `runtime/main.c`, который вызывает `compiled_fn`.

### 3. Объясните роль `flex`/`re2c` и `bison` в проекте

В этом проекте используется `flex` и `bison`. `re2c` здесь не используется.

Роль `flex`:

- берет файл `src/lexer.l`;
- генерирует лексер;
- лексер читает текст и возвращает токены парсеру.

Примеры токенов:

- `RETURN`
- `IF`
- `IDENT`
- `INTEGER`
- `EQ`
- `AND`

Роль `bison`:

- берет файл `src/parser.y`;
- генерирует парсер;
- парсер получает токены от лексера;
- проверяет, соответствует ли поток токенов грамматике;
- строит AST.

Итого:

- `flex` отвечает за уровень символов и токенов;
- `bison` отвечает за уровень структуры программы.

### 4. Зачем нужен файл с кодом абстрактного синтаксического дерева

Файл `src/ast.hpp` нужен, чтобы хранить программу в удобной внутренней форме.

Он решает сразу несколько задач:

- дает единое представление программы после парсинга;
- отделяет синтаксический разбор от семантики;
- отделяет семантику от генерации кода;
- позволяет всем следующим этапам работать не с текстом, а со структурой.

Например:

- парсер создает `BinaryExpr`, `IfStmt`, `FunctionDecl`;
- семантика проверяет эти узлы;
- codegen проходит по этим же узлам и строит LLVM IR.

Если бы AST не было, пришлось бы:

- либо генерировать код прямо внутри парсера;
- либо повторно анализировать исходный текст на следующих этапах.

Оба варианта заметно хуже и сложнее в поддержке.

### 5. Как реализован приоритет арифметических операций

Приоритет реализован структурой грамматики в `src/parser.y`.

Уровни идут сверху вниз так:

- `assignment`
- `ternary`
- `logical_or`
- `logical_and`
- `equality`
- `relational`
- `additive`
- `multiplicative`
- `unary`
- `primary`

Из-за этого:

- `*`, `/`, `%` сильнее, чем `+` и `-`;
- унарные операции сильнее бинарных;
- скобки в `primary` могут изменить обычный порядок.

Пример:

```c
2 + 3 * 4
```

разбирается как:

```text
2 + (3 * 4)
```

а не как:

```text
(2 + 3) * 4
```

Левая ассоциативность достигается леворекурсивными правилами:

```bison
additive
    : additive '+' multiplicative
    | additive '-' multiplicative
```

Поэтому:

```c
10 - 3 - 2
```

разбирается как:

```text
(10 - 3) - 2
```

### 6. Что поддерживает семантический анализ в проекте

Семантический анализ реализован в `src/sema.cpp`.

Он поддерживает:

- проверку повторного объявления функций;
- обязательное наличие `compiled_fn`;
- проверку сигнатуры `compiled_fn`;
- области видимости переменных;
- проверку повторного объявления переменной в одном scope;
- использование только объявленных переменных;
- проверку типов инициализаторов;
- проверку типов присваиваний;
- проверку типов арифметических операций;
- проверку типов сравнений;
- проверку типов логических операций;
- проверку типа условия в `if`;
- проверку типа условия в `for`;
- проверку типа условия в `do/while`;
- проверку типа возвращаемого значения;
- проверку вызовов функций;
- проверку количества аргументов;
- проверку типов аргументов;
- проверку совпадения типов ветвей тернарного оператора;
- вычисление `inferredType` для выражений.

То есть семантика здесь уже довольно полноценная для учебного мини-языка.

### 7. Как обрабатываются ошибки синтаксиса

Ошибки синтаксиса обрабатываются через `yyerror()` в `src/parser.y`.

Когда Bison обнаруживает некорректную последовательность токенов:

- вызывается `yyerror(const char *message)`;
- сообщение складывается в `g_parse_errors`;
- в текст ошибки добавляются строка и столбец.

Позиции ошибки берутся из `yylloc`, который заполняется лексером через `YY_USER_ACTION` в `src/lexer.l`.

Дальше `src/driver.cpp` после `yyparse()` проверяет:

- код возврата парсера;
- наличие `g_parse_errors`;
- наличие построенной программы.

Если что-то пошло не так, драйвер печатает:

```text
parse error: line X, column Y: ...
```

и завершает работу.

Важно:

- здесь есть хорошая диагностика с координатами;
- но нет сложного recovery-механизма, который продолжает разбор после ошибки;
- поэтому проект ориентирован на понятную остановку с сообщением, а не на восстановление нескольких ошибок за один проход.

### 8. В чем смысл LLVM IR и общие принципы его построения

LLVM IR нужен как промежуточный язык между AST и машинным кодом.

Зачем это полезно:

- не нужно вручную писать backend сразу из AST в MIPS-инструкции;
- проще проверять корректность генерации;
- можно использовать готовую инфраструктуру LLVM;
- можно разделить frontend языка и backend архитектуры.

Общий принцип построения LLVM IR в проекте такой:

1. Для каждой функции создается LLVM `Function`.
2. Для параметров и локальных переменных создаются `alloca`.
3. Операции AST переводятся в LLVM-инструкции.
4. Для ветвлений и циклов создаются `BasicBlock`.
5. Для слияния значений из нескольких путей используются `PHI`-узлы.
6. Затем модуль проверяется verifier-ом LLVM.
7. После этого LLVM backend выпускает объектный файл.

Примеры соответствия:

- `a + b` -> `CreateAdd`
- `a * b` -> `CreateMul`
- `a < b` -> `CreateICmpSLT`
- `if` -> `CreateCondBr` и несколько basic blocks
- `x ? y : z` -> ветвление и `PHI`
- `&&` и `||` -> short-circuit через блоки и `PHI`

Смысл LLVM IR в этом проекте:

- это удобный мост между учебным языком и MIPS;
- он делает архитектуру компилятора чище;
- он позволяет не смешивать синтаксис языка с деталями целевой платформы.

## 19. Пять маленьких экспериментов, чтобы руками увидеть работу компилятора

Ниже пять коротких изменений по возрастанию сложности. Идея простая: вы меняете одну строку, запускаете несколько команд и сразу видите, что изменилось в поведении программы или в диагностике компилятора.

### Эксперимент 1. Изменить входной аргумент

Файл: `runtime/main.c`

Замените:

```c
int64_t result = compiled_fn(10);
```

на:

```c
int64_t result = compiled_fn(5);
```

Запустите:

```bash
mipsel-linux-gnu-gcc runtime/main.c build/05_do_while_sum.o -o build/05_do_while_sum.mips
qemu-mipsel -L /usr/mipsel-linux-gnu build/05_do_while_sum.mips
```

Ожидаемый результат:

```text
10
```

Почему так: функция теперь считает сумму чисел от `0` до `4`.

### Эксперимент 2. Изменить условие цикла

Файл: `tests/05_do_while_sum.mc`

Замените:

```c
} while (i < arg);
```

на:

```c
} while (i <= arg);
```

Запустите:

```bash
./build/mini_cc tests/05_do_while_sum.mc -o build/05_do_while_sum.o --emit-ir build/05_do_while_sum.ll
mipsel-linux-gnu-gcc runtime/main.c build/05_do_while_sum.o -o build/05_do_while_sum.mips
qemu-mipsel -L /usr/mipsel-linux-gnu build/05_do_while_sum.mips
```

Если в `runtime/main.c` осталось `compiled_fn(10)`, ожидаемый результат:

```text
55
```

Почему так: цикл теперь включает и число `10`.

### Эксперимент 3. Изменить шаг цикла

Файл: `tests/05_do_while_sum.mc`

Замените:

```c
i = i + 1;
```

на:

```c
i = i + 2;
```

Запустите те же три команды, что и в эксперименте 2.

Если условие снова `i < arg`, а в `runtime/main.c` стоит `10`, ожидаемый результат:

```text
20
```

Почему так: суммируются только `0 + 2 + 4 + 6 + 8`.

### Эксперимент 4. Заменить всю логику функции на простую формулу

Файл: `tests/05_do_while_sum.mc`

Сделайте файл таким:

```c
int compiled_fn(int arg) {
    return arg * 3;
}
```

Запустите:

```bash
./build/mini_cc tests/05_do_while_sum.mc -o build/05_do_while_sum.o --emit-ir build/05_do_while_sum.ll
mipsel-linux-gnu-gcc runtime/main.c build/05_do_while_sum.o -o build/05_do_while_sum.mips
qemu-mipsel -L /usr/mipsel-linux-gnu build/05_do_while_sum.mips
```

При `compiled_fn(10)` ожидаемый результат:

```text
30
```

Почему так: вы убрали цикл и заменили его обычным выражением.

### Эксперимент 5. Специально вызвать semantic error

Файл: `tests/05_do_while_sum.mc`

Замените:

```c
sum = sum + i;
```

на:

```c
sum = missing + i;
```

Запустите только компиляцию:

```bash
./build/mini_cc tests/05_do_while_sum.mc -o build/05_do_while_sum.o --emit-ir build/05_do_while_sum.ll
```

Ожидаемый результат: объектный файл не соберется, а компилятор напечатает ошибку про необъявленную переменную `missing`.

Это хороший эксперимент, потому что он показывает разницу между:

- корректной программой, которая доходит до LLVM IR и объектного файла;
- и программой, которую останавливает семантический анализ.
